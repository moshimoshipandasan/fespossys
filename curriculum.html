👀 単元概要

本単元は、文部科学省「高等学校学習指導要領（平成30年告示）解説 情報編 情報Ⅰ」※1で示された「情報社会の問題解決」「コミュニケーションと情報デザイン」「コンピュータとプログラミング」の3領域を、文化祭模擬店向け POS システム「FesPos」を題材に実践的に学ぶ6時間構成の授業案です。

単元名

「文化祭を支える情報デザインとプログラミング」

対象

高等学校 情報Ⅰ（普通科／情報系列）

学習時間

50分 × 6時間（プロジェクト型学習）

学習形態

グループワーク・協働探究・発表型

題材の特徴
  • 「祭りを支えるデジタル化」という身近な課題を設定し、情報システムの社会的価値を実感させます。
  • 既存システム（FesPos）を活用しながら、分析→設計→実装→テスト→改善の一連を追体験します。
  • スプレッドシート／Apps Script／Web UI を横断的に扱い、情報Ⅰで扱うデータ活用とプログラミングを結び付けます。

🎯 情報Ⅰの資質・能力との対応

学習指導要領解説に基づき、3つの領域と FesPos 活動の対応を整理しました。

領域（情報Ⅰ） 学習指導要領の学習内容 FesPos プロジェクトでの学び
問題解決情報社会の問題解決 「情報システムを社会の課題解決に活用する過程を理解し、影響を考察する。」※1 p.116-118 文化祭の売上管理・在庫課題を洗い出し、POS システムの導入効果や停止時のリスクを議論。データ活用の意義を検討。
情報デザインコミュニケーションと情報デザイン 「利用者の状況や目的に応じて情報を設計し、効果的に提示する。」※1 p.110-113 POS 画面やダッシュボードをレビューし、操作性・視認性を評価。改善提案をワイヤーフレームや紙面で表現し共有。
プログラミングコンピュータとプログラミング 「プログラムを構造的に捉え、アルゴリズムを設計・実装・検証する。」※1 p.120-121 Apps Script の処理構造（販売記録・在庫更新）を読み解き、フローチャート化。テストケース設計とコード改良案の検討を行う。

📌 単元のねらい

  1. 情報システムが社会にもたらす価値と課題を、具体的な事例とデータで説明できる。（知識・理解）
  2. 利用者視点で要件を整理し、情報の構造と画面デザインを協働で設計できる。（思考・判断・表現）
  3. アルゴリズムを図や擬似コードで表現し、テスト観点を踏まえて改善提案を作成・発表できる。（技能・表現）
評価観点との関係

各学習活動は「情報社会の問題解決の能力」「コミュニケーションへの関心・意欲・態度」「プログラミングの技能」の観点に対応します。評価規準は後述の表を参照してください。

🗓 授業構成（全6時間）

1コマ50分の想定。授業ごとの主な活動と評価の着眼点を示します。

第1時 情報システムの価値

文化祭 POS を題材に課題を発見する

  • 導入：文化祭での課題を想起し、POS 導入の目的を共有
  • 活動：FesPos の概要資料を読み、情報の流れを図示
  • 振り返り：導入効果と停止時の影響をグループ発表

評価：課題発見と社会的影響の洞察

第2時 要件定義

利用者視点で機能要件を洗い出す

  • 既存メニュー（🎪 POSシステム設定）を分析し、現状機能を整理
  • 利用者（店長・会計担当・在庫担当）のニーズを付箋で可視化
  • ユーザーストーリーマップを作成し、優先順位を検討

評価：利用者視点の要件整理

第3時 情報デザイン

UI/UX とセキュリティを考える

  • Web POS 画面を分解し、情報設計の工夫をカードにまとめる
  • セキュリティ要件（認証・ログ・権限）を追加する理由を討議
  • 改善案をワイヤーフレームや紙面デザインで提示

評価：情報の整理と根拠をもった改善提案

第4時 アルゴリズム理解

コードリーディングで処理を理解する

  • processTransaction 等の Apps Script を読み解き処理手順を要約
  • 在庫更新ロジックをフローチャートまたは擬似コード化
  • 関数間の連携（モジュール構造）をシンプルな図にまとめる

評価：アルゴリズムの可視化・説明力

第5時 テスト設計

品質保証の視点でテストを計画する

  • 正常系／異常系のテストケースを作成し、入力・手順・期待結果を整理
  • if 分岐を洗い出し、網羅性（条件網羅）を自己チェック
  • 他グループとテストケースを相互レビュー

評価：論理的なテスト設計と改善提案

第6時 振り返りと発表

改善提案と学びの成果を共有する

  • システム改善案（例：割引処理、在庫補充通知）を短い企画書にまとめる
  • プロジェクト全体を振り返り、作業分解（WBS）と役割を整理
  • 改善提案とテスト結果を発表し、相互評価とふり返りカードを記入

評価：成果物の構造化と振り返り

🛠 探究プロジェクトの進め方

準備物（教師）

  • FesPos テンプレートシートのコピー（班ごとに配布）
  • README／manual／curriculum の印刷・共有リンク
  • 付箋・模造紙・マグネットなどのアイデア整理ツール

学習者の役割例

  • プロジェクトマネージャー：進行と振り返りを担当
  • UX デザイナー：ワイヤーフレームと情報整理
  • データ／テスト担当：テストケースと改善提案の検証

成果物

  • 情報の流れ図・ユーザーストーリーマップ
  • UI 改善案／フローチャート／テストケース表
  • 改善企画書（A3 1枚）と発表スライド 3 枚程度
ICT 活用の工夫
  • Google スプレッドシートで共同編集し、コメント機能でレビュー。
  • Jamboard やスクリーンショットで UI 評価を共有。
  • clasp や Apps Script エディタのログを使って動作確認を補助。

🔗 教科横断と主体的・対話的で深い学び

他教科との連携例

  • 数学：データ分析（売上推移の統計処理）
  • 家庭科／商業：販売計画・食材管理との連携
  • 国語・英語：操作マニュアルや案内文の多言語化

探究的な学びの深化

  • 文化祭当日を想定したロールプレイで、システム活用の課題を検証
  • 地域イベントや SDGs（食品ロス vs. 適正在庫）への展開を考える
  • AI（Gemini 連携）の活用価値と倫理を討議する

学習評価のフィードバック

  • スプレッドシートコメントでリアルタイムに指導・助言
  • Rubric（三段階）で観点別評価を可視化し自己評価と比較
  • 成果物を文化祭実行委員会等に共有し、社会的承認を得る
